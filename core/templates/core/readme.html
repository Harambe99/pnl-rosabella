1. What this is

A web app that tracks the P&L for a TikTok Shop. Replaces the old Google Sheets version (date bugs, formula drift, no real dedup).

Every line on the P&L comes from one of three places:

  1. An uploaded TikTok export file.
  2. The COGS table (per-SKU costs).
  3. The Monthly Inputs tab (manual entries for things with no export).

The aggregator combines them and shows you Daily and Monthly views.

4. The 7 file imports

Re-uploading the same file is always safe — duplicates get skipped (Settlement merges multi-SKU rows by summing). Every upload logs to the History page. On Settlement uploads, the Reports tab parents are cross-checked against per-order sums; any drift >$1 lands in the History note as a "Drift vs Reports tab" alert.

6. Ad Discounts ledger — FIFO ad-account reconciliation

A separate page (/ad-discounts/) reconstructs the daily TikTok ad-account balance from your Transaction export and answers: was today's spend funded by discounted credit, and at what rate, or was it card-billed at full price?

How it works

  1. Credit layers — every "Increase balance" row in the Transaction export's Others sheet becomes a credit layer at the default TBSM rate (6%). Promotions-sheet rows default to free (100% discount) unless you tag a date as agency-purchased at a specific rate.
  2. FIFO drawdown — daily ad spend draws against the oldest credit layer first. When the queue empties, the rest of the day's spend is full-price card-billed.
  3. Verification — modeled full-price spend should approximately equal actual card "Bill payment" totals per month. Significant drift = something needs to be tagged.

Setup (one-time)

  • Upload the widest-range Transaction export from Ads Manager (covers TBSM loads, promos, card charges)
  • Upload the widest-range Campaign overview (Ad Spend) so spend history matches transaction history
  • Set the Opening balance — for our shop, $0 on 2025-06-11 (one day before the first TBSM load). This is provable: card charges between 2025-06-05 and 2025-06-17 mean the balance was $0 in that window.
  • Add Agency Promo Tags for any non-default rate (e.g. KDMT 10% on May 8/10/29 2026)
  • Tick "Feed P&L" to let the engine populate the Less: TBSM Savings and Less: TT Promo Credits lines on Daily/Monthly P&L. Defaults off until verified.

Ongoing maintenance

  • Re-upload Transaction + Campaign overview periodically (engine auto-recomputes on each upload)
  • When a new agency-purchased promo deal happens at a non-default rate, add an Agency Promo Tag
  • Agency Invoices section is informational only — engine doesn't read invoices

7. COGS — SKU cost lookup

The COGS page is a table of SKU ID → cost-per-order. When Manage Orders is imported, each line's COGS is computed as quantity × COGSItem.cogs_per_order and stored on the Order row.

Initial seed: the management command seed_cogs runs on every deploy (in build.sh) — it creates COGS rows for known SKUs with default costs. It does NOT overwrite costs you've edited on the page.

To update a cost: edit the value inline on the COGS page and click Save. New imports will use the new value; old imported orders keep the COGS they were imported with. To recompute old orders with the new costs, wipe Orders and re-import (see Admin section).

Adding a new SKU: add a row to core/management/commands/seed_cogs.py and push — Render runs seed_cogs on every deploy. Or insert directly via Django shell.

8. P&L line items — what powers each one

Each line below tells you four things: where the number comes from, the exact formula, why that formula (not something else), and what to watch out for.

REVENUE

Gross Sales Manage Orders
From: the SKU Subtotal Before Discount column of the Manage Orders CSV, grouped by the day the order was created (Created Time column), with any row where Order Status is "Canceled" excluded.
Formula: sum(SKU Subtotal Before Discount) for each created-date, excluding canceled.
Why this column: it's the only one that captures full retail revenue before ANY discount. Every other revenue column in the CSV is post-discount or partial — this is the only "true gross" number.
Watch: "Canceled" is the only excluded status. Refunded orders DO still count here — they were real sales that later got refunded. The refund hits the Less: Refunds line separately.
Less: Promos & Discounts Manage Orders
From: the SKU Seller Discount column of the same Manage Orders CSV, grouped by created-date.
Formula: -1 × sum(SKU Seller Discount). Stored negative so it visually subtracts.
Why only seller discount: TikTok separates seller-funded discounts (you paid for them) from platform-funded discounts (TikTok paid for them). We only count what you paid — TikTok's discounts don't reduce your revenue, they reduce TikTok's books.
Scope: this line covers SKU-level product price discounts only. Shipping-related promos (e.g., "free shipping over $30") are NOT in this line — they show up indirectly via the shipping-cost lines below (since the customer didn't pay shipping, the cost burden falls entirely on you). See the Seller Shipping and FBT Fulfillment Fee sections.
GMV CALC
From: computed from Manage Orders (Method A).
Formula: GMV = Gross Sales + Less: Promos & Discounts
Why computed not overridden: the rest of the P&L is accrual-based (orders attributed by Created Time). Using Gross − Discount keeps the displayed math self-consistent (the components add up to GMV). TikTok's Shop Analytics GMV is shown as a separate reference line for comparison but does NOT drive Net Revenue.
GMV (TikTok Analytics — reference) Shop Analytics
From: Shop Analytics XLSX, GMV column. Published by TikTok.
Formula: TikTok's value as-is, not computed.
Reference only — does NOT feed Net Revenue. TikTok applies internal calculation rules (excludes certain order types, fraud filtering) so this number typically differs slightly from our computed GMV. Useful for "what does my dashboard show?" comparisons.
Watch: if Shop Analytics isn't uploaded for a month, this line shows blank for that month. The Net Revenue still computes correctly via Method A.
Less: Refunds Settlement
From: two columns of the Settlement XLSX added together — Gross sales refund + Seller discount refund — on rows where Type = "Order".
Formula: sum(Gross sales refund) + sum(Seller discount refund), grouped by the order's created date (not the date the refund was settled).
Why both columns: when an order was originally sold, TikTok recorded gross sales AND a seller discount as separate entries. When that order is refunded, both entries get reversed.

Example: sold for $20 with $10 seller discount → customer paid $10, you received $10.
When refunded: Gross sales refund = -$20, Seller discount refund = +$10, net effect on your wallet = -$10.

Using only Gross sales refund would record -$20 (you'd be over-refunded on paper). Using only Seller discount refund would record +$10 (sign flipped, wrong direction). Summing the two gives the true cash impact.
Watch: refunds are attributed to the day the original order was created. A May refund for an April order shows on the April day. Re-uploading newer Settlement files will keep adding late refunds to old days.
NET REVENUE CALC
Formula: NET REVENUE = GMV + Less: Refunds (refunds stored negative, so this subtracts).
Why this is the headline: GMV is what you sold; Net Revenue is what stayed sold after returns. This is the number that flows into Gross Profit.

COGS & FULFILLMENT

COGS COGS table Manage Orders
From: at import time, every Manage Orders row is matched on SKU ID to the COGS table to get a per-unit cost, then stored as cost × quantity on the Order row in the database.
Formula: -1 × sum(Order.cogs) by created-date, excluding canceled orders.
Why frozen at import: COGS represents what you ACTUALLY paid the supplier when that order shipped. Editing the COGS table today shouldn't retroactively change the cost of last month's orders — that would distort historical reporting.
Watch: if a SKU isn't in the COGS table, its cost is $0 (no error). After every Manage Orders upload, check the History tab — unmapped SKUs are listed in the import notes.
FBT Fulfillment Fee Settlement
From: Settlement XLSX, FBT fulfillment fee column, only on rows where Type = "Order".
Formula: sum(FBT fulfillment fee) by order created-date.
What this is: the bundled fee TikTok charges for picking, packing, and shipping an FBT order. Settlement records it per-order so we can attribute it to the day the order was placed.
Watch: this is ALSO present (per-SKU) in the Logistics Cost Overview file. Do not import it from there too — it would double-count.
FBT Fulfillment Reimbursement Settlement
From: Settlement, FBT fulfillment fee reimbursement column, on Order-type rows.
Formula: sum(reimbursement) by order created-date. Stored positive (it's money back to you).
When this fires: TikTok charged you the FBT Fulfillment Fee but a fulfillment problem occurred (lost package, damaged, etc.) and they refunded part or all of it.
Shipping section — overview
TikTok's Shipping parent line on the Reports tab (e.g., -$77,565.91 for May 2026) is the sum of seven sub-components. We surface each of them as its own line below so the P&L matches the Reports tab exactly when summed, and so each piece is independently visible. Older versions of this app summed some of these into composite lines (Seller Shipping, TT Shipping Subsidy (net)) — both have been retired because they overlapped with the components and double-counted small amounts.
TT Shop Shipping Incentive Settlement
From: Settlement, TikTok Shop shipping incentive column, on Order-type rows.
Formula: sum by order created-date.
What it is: when an order qualifies for free shipping (customer paid $0 for shipping), TikTok credits the seller the equivalent shipping fee so the seller appears whole. May 2026: +$95,400.72.
Pair: almost always cancelled per-row by Customer Shipping Fee Offset. Net seller impact ≈ $0.
Shipping Fee Subsidy Settlement
From: Settlement, Shipping fee subsidy column, on Order-type rows.
Formula: sum by order created-date.
What it is: a separate subsidy program (typically for seller-shipped orders). Usually $0 — only fires when a specific seller subsidy program is active. May 2026: $0.
Customer Shipping Fee Offset Settlement
From: Settlement, Customer shipping fee offset column, on Order-type rows.
Formula: sum by order created-date. Negative.
What it is: TikTok's clawback that mirrors either the Shipping Incentive OR the Customer-Paid Shipping Fee — it's TikTok keeping the shipping money to pay the carrier. Per TikTok docs: "typically used to offset either TikTok Shop's shipping incentive or customer-paid shipping fees, resulting in a net charge of $0 to you." May 2026: −$96,508.91.
Why it doesn't always wash: if the seller has opted into seller-funded shipping for FBT-fulfilled products, the offset hits with no matching credit. That's the source of the ~$1K residual after the incentive/offset cancel out.
Customer-Paid Shipping Fee Settlement
From: Settlement, Customer-paid shipping fee column, on Order-type rows.
Formula: sum by order created-date. Positive.
What it is: the shipping fee the customer actually paid (orders that didn't qualify for free shipping). On a fully-free-shipping shop this column is mostly $0; with a $30 free-shipping threshold, sub-threshold orders contribute. May 2026: +$3,432.80.
Customer-Paid Shipping Refund Settlement
From: Settlement, Customer-paid shipping fee refund column, on Order-type rows.
Formula: sum by order created-date. Negative.
What it is: the seller-side reversal when a customer's shipping payment is refunded (returns or cancellations on paid-shipping orders). May 2026: −$158.57.
Cost to Ship to FBT Monthly Input
From: the cost_ship_to_fbt field on Monthly Inputs (manual entry).
Formula: -1 × (monthly_value ÷ days_in_month), applied to every day of the month.
Why manual: when YOU arrange inbound shipping (your 3PL ships your stock to FBT), it doesn't appear in any TikTok export. TikTok's "Inbound Shipping Fee" in FBT Billing only covers the case where TikTok arranged the inbound — usually $0 for self-shipped stock.
Watch: sum your monthly inbound shipping invoices and enter the total. Daily P&L will show ~total/30 each day; monthly view will show the real monthly total.
Cost to Ship to Customer Seller Shipping CSV
From: SellerShipmentCost.postage rows imported from the Seller Shipping CSV (3PL postage report), grouped by shipped_date.
Formula: -1 × sum(postage) for each shipped date.
Why ship-date not order-date: the cost hits your account the day the package leaves the warehouse, not the day the order was placed. With ~1-day median order-to-ship lag, daily P&L shows yesterday's orders' shipping on today's costs — close enough that we don't try to back-attribute.
Watch: if no Seller Shipping CSV is uploaded for the month, this line is $0 (no manual fallback). Multi-SKU shipments repeat the postage value across rows in the CSV; the importer dedups on shipment_number so postage is counted once.
Logistics Reimbursement Settlement
From: Settlement rows where Type = "Logistics reimbursement". Value pulled from the Adjustment amount column.
Formula: sum(adjustment_amount) for those rows, by order created-date. Positive (money back to you).
What this is: TikTok reimbursing logistics-side mistakes — wrong address routing fees they shouldn't have charged, etc.
FBT Hub Placement / Storage / Inbound Shipping / Incidents / Booking N/C / Routing N/C / Outbound No-Show / Delayed Response / Disposal / Return Shipping (VAS) / Return-to-Seller Handling / Inbound Return Operation FBT Billing
From: the FBT Billing XLSX ("Logistics Cost Overview" export). Each row in the file has a line description (column B) and an amount (column C). The importer matches the line description against a 12-entry mapping (FBT_LINE_TO_FIELD in importers.py) and writes the amount to the matching MonthlyInput field.
Formula per line: -1 × (MonthlyInput.fbt_X ÷ days_in_month), per day of the month.
Why monthly: these fees are billed once a month, not per-order. Flat-spreading across days gives a daily approximation without pretending we know exactly when each fee was incurred within the month.
Watch: re-uploading FBT Billing for the same month OVERWRITES all 12 fields. If you make a manual edit between uploads, it gets clobbered.
FBT Warehouse Compensation Settlement
From: Settlement rows where Type = "FBT warehouse compensation", value from Adjustment amount.
What this is: TikTok comping you for warehouse-side losses (lost inventory, damaged stock not your fault, etc.). Different from FBT Fulfillment Reimbursement, which is per-order.

Platform Fees

Referral Fee Settlement
From: Settlement, Referral fee column, on Order-type rows.
Formula: sum(Referral fee) by order created-date.
What this is: TikTok's commission for being the marketplace — typically a fixed % of GMV. The largest single platform fee.
Refund Admin Fee Settlement
From: Settlement, Refund administration fee column.
What this is: a small per-refund admin charge TikTok takes when processing returns. Scales with refund volume.
Campaign Service Fee Settlement
From: Settlement, Campaign service fee column.
What this is: fee for participating in TikTok promo campaigns. Often $0 unless you opted into a campaign that month.
Violation Fee Settlement
From: Settlement rows where Type = "Violation fee (settlement fee)" (matched leniently — TikTok sometimes uses a Chinese parenthesis character). Value from Adjustment amount.
What this is: penalties for policy violations (listing rule breaks, late fulfillment streaks, etc.). Should generally be $0 — investigate if it's not.
TikTok Shop Reimb Settlement
From: Settlement rows where Type = "TikTok Shop reimbursement". Value from Adjustment amount. Positive.
What this is: compensation TikTok pays you for various platform-side issues — refund-protection refunds, dispute resolutions, etc.
Rebate Settlement
From: Settlement rows where Type = "Rebate". Value from Adjustment amount. Positive.
What this is: volume-based or program-based rebates TikTok pays back to qualifying sellers.
Co-funded Promotion (seller-funded) Settlement
From: Settlement, Co-funded promotion (seller-funded) column on Order-type rows.
What this is: your share of a co-funded TikTok-and-you promo. TikTok funds part, you fund part — this column captures your contribution per order. Stored negative (cost to you).
GROSS PROFIT CALC
Formula: Net Revenue + all 28 cost/fee/reimbursement lines above (most are negative, some are positive reimbursements that increase profit).
What this represents: profit after every direct cost of selling and fulfilling the order — but before marketing and overhead. This is the "unit economics" line.

MARKETING

Ad Spend — Direct to TikTok (cash) Ad Spend
From: Ad Spend XLSX (Campaign Overview), Cost column by day.
Formula: -1 × AdSpendDay.cost for each date.
What this is: the cash you paid TikTok for ads that day. The headline marketing cost.
Watch: re-uploading Ad Spend OVERWRITES per-date values. If TikTok adjusted a daily spend later (rare but happens), the new value wins.
Less: TT Promo Credits Monthly Input
From: the tt_promo_credits field on Monthly Inputs (manual).
Formula: +1 × (monthly_value ÷ days_in_month), per day. Positive sign — reduces the negative Ad Spend.
Why manual: TikTok shows promo credit balances in Ads Manager but doesn't include them in the Campaign Overview export. Until a parser is added, enter monthly. Often $0.
Less: TBSM Savings (6%) currently $0
From: hardcoded to $0 in the aggregator.
History: previously auto-calculated as 6% × Ad Spend. Removed because the real TBSM (TikTok Brand Spend Match) rate varies and isn't in any current export.
Impact: Total Ad Spend reads ~6% higher than the true net cost if TBSM is active. Net Profit is slightly understated. Wire in a real source when found.
Total Ad Spend CALC
Formula: Ad Spend (negative) + TT Promo Credits (positive). TBSM contributes 0 for now.
Platform (Affiliate Commission) Settlement
From: Settlement, four columns added together — Affiliate Commission + Affiliate partner commission + Affiliate Shop Ads commission + Affiliate Partner shop ads commission. On Order-type rows.
Formula: sum of all four, by order created-date.
Why four columns: TikTok separates direct affiliates, partner-network affiliates, Shop Ads-driven affiliates, and partner Shop Ads. Each goes into its own settlement column. We sum them because they're all "you paid a creator a commission" from a P&L perspective.
Watch: if "Affiliate Commission" is ever a TOTAL of the other three rather than a separate bucket, we'd double-count. Manually reconcile one settlement file against a creator-payout report to confirm.
Off-Platform (1% method) Monthly Input
From: the off_platform_1pct field (manual).
What this is: 1% commission paid to off-platform creators (creators driving traffic from outside TikTok). Calculated by your team — there's no TikTok export for it.
Monthly Retainers Monthly Input
From: monthly_retainers (manual). Sum of fixed creator/agency retainers for the month.
Outsourced Agency Monthly Input
From: creatify field (manual). The DB column name is legacy; the displayed label is generic so any agency works.
Why generic naming: originally specific to Creatify. Renamed for reuse with any future external agency without a migration.
TOTAL MARKETING CALC
Formula: Total Ad Spend + Affiliate Commission + Off-Platform + Monthly Retainers + Outsourced Agency. All negative.

SG&A

Team Spend Monthly Input
From: team_spend (manual). Payroll + contractors for the month.
Software & Tools Monthly Input
From: software_tools (manual). Monthly SaaS bill total (Shopify, ChatGPT, Slack, etc.).
Other G&A Monthly Input
From: other_ga (manual). Catch-all for general & admin costs that don't fit another bucket.
Chargebacks Settlement
From: Settlement rows where Type = "Chargeback". Value from Adjustment amount. Stored negative.
What this is: customer-initiated payment disputes (credit card chargebacks) that TikTok absorbed from your wallet. Attributed to the original order's created date.
Unclassified Adjustments Settlement
From: Settlement rows whose Type field doesn't match any of the known mappings in TYPE_TO_FIELD. Value from Adjustment amount.
Why this exists: safety net. If TikTok introduces a new settlement row type we haven't mapped, the money lands here instead of being lost. The new type name is logged to ImportLog notes.
Watch: if this line grows, check the History tab notes. Decide whether to add the new type to TYPE_TO_FIELD in importers.py (mapping it to the right field) or leave it in unclassified.
TOTAL G&A CALC
Formula: Team Spend + Software & Tools + Other G&A + Chargebacks + Unclassified Adjustments.

Bottom line

NET PROFIT CALC
Formula: GROSS PROFIT + TOTAL MARKETING + TOTAL G&A (Marketing and G&A are already negative).
What this is: the bottom-line number — what's left after every direct cost, every marketing dollar, and every overhead expense. Negative for now; goal is positive.

9. How the numbers roll up

  1. Daily P&L (compute_daily_pnl in aggregator.py) for a date range:
    1. Aggregate Order rows by created_date → Gross / Promos / COGS.
    2. Aggregate SettlementRow by order_created_date → all the settlement-driven lines.
    3. Override GMV with AnalyticsDay.gmv if present.
    4. Set raw Ad Spend from AdSpendDay.cost (negated).
    5. For each Monthly Input field with an overlay mapping, set the day's value to sign × monthly / days_in_month.
    6. Override Cost to Ship to Customer with real daily sums from SellerShipmentCost if any rows exist for that month.
    7. Compute derived rows: GMV (fallback), Net Revenue, Gross Profit, Total Ad Spend, Total Marketing, Total SG&A, Net Profit.
  2. Monthly P&L (compute_monthly_pnl) loops months Jan-Dec, runs compute_daily_pnl for each, sums daily values per row.
  3. Render: views.py builds row-by-row data using PNL_ROW_LAYOUT (the canonical row order + types: section / sub / row / total / blank), passes to templates.

  2. Make changes locally.
  3. If you changed models: python manage.py makemigrations and commit the migration file.

11. Admin / data operations

Wipe imported data

The /wipe/ endpoint clears imported data without touching COGS or Monthly Inputs. POST only, protected by the shared-password middleware.

# Wipe everything
curl -X POST -b "sessionid=<your_session>" https://pnl-rosabella.onrender.com/wipe/

# Wipe a single source
curl -X POST "https://pnl-rosabella.onrender.com/wipe/?what=orders"
# Options: orders | settlement | analytics | ad_spend | seller_shipping | all

To wipe COGS or Monthly Inputs, use Django admin or shell — the wipe endpoint deliberately won't touch those.

Adding a new admin user

There are no per-user accounts. Everyone shares one password (APP_PASSWORD env var on Render). To rotate: change the env var in Render dashboard → service restarts automatically → tell users the new password.

Database access

Render dashboard → Postgres service → connect via psql with the internal/external connection string. Tables are prefixed core_ (e.g., core_order, core_settlementrow).

13. Known gotchas & design decisions

  • Settlement files MUST be parsed with pandas, not openpyxl read-only. openpyxl read-only mode has a bug that returns only ~5 rows from the iterator. pandas is also more memory-efficient (~175 MB vs 500 MB+).
  • Date format is auto-detected per file. Shop Analytics uses DD/MM/YYYY, Ad Spend uses MM/DD/YYYY, Manage Orders uses MM/DD/YYYY with AM/PM. The _to_date helper in importers.py pre-scans the column for any day > 12 to disambiguate.
  • GMV does NOT include refunds. Historical bug: refunds used to be added to GMV, double-counting in Net Revenue. Fixed: GMV = Gross + Promos (post-discount, pre-refund). Net Revenue = GMV + Refunds.
  • Settlement rows attribute to Order Created Date, not Settlement Date. This is what makes late settlements (June file containing April orders) populate April rows correctly.
  • Seller Shipping CSV has duplicated postage across multi-SKU shipments. Each line of a 5-SKU shipment repeats the same postage value. The importer dedups on shipment_number so only one is counted.
  • Only ~50% of Seller Shipping reference_numbers match Order IDs in the export. The other 50% are RESHIP, sample, or out-of-window orders — all real costs. Don't try to "fix" this by tightening the match; the postage total is the trusted number.
  • Section / sub / blank rows render empty cells, NOT dashes. Dashes only appear on data rows with value 0 or null.
  • The /wipe/ endpoint is CSRF-exempt so it can be hit via curl. It's still gated by the shared-password middleware.
  • Single gunicorn worker, 4 threads, 900s timeout. The threads + long timeout are needed because large file imports (37 K+ settlement rows) take minutes; a single worker keeps memory bounded.
  • TBSM 6% auto-calc was removed. Was previously hardcoded as Ad Spend × 0.06. Now zero — TBD how to source the real value (likely a new MonthlyInput field).

If you find something in the code or docs that contradicts the live behavior, the code is the source of truth — please update this README so the next person isn't misled.
