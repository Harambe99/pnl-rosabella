1. What this is

A web app that tracks the P&L for a TikTok Shop. Replaces the old Google Sheets version (date bugs, formula drift, no real dedup).

Every line on the P&L comes from one of three places:

  1. An uploaded TikTok/3PL export file.
  2. The COGS table (per-SKU costs).
  3. The Monthly Inputs tab (manual entries for things with no export).

The aggregator combines them, applies the correct attribution methodology per line, and shows you Daily and Monthly views. Every displayed number reconciles to the penny with the downloadable XLSX export.

4. Attribution methodology — how each line dates its values

This is the most important section. Different P&L lines attribute their values to different days depending on when the underlying business event happened. Confused methodology = wrong-month reporting = P&L drift vs TikTok / accounting.

Four methodologies are in use across the P&L, each chosen deliberately for that line's economics. All four are anchored to Lindsay + Jack's June 25-30, 2026 spec, with FBT reverted to services-month on Jack's July 1 request.
MethodologyApplied toAnchoring dateWhy
Statement Date Gross Sales, Promos & Discounts, Refunds, COGS, all Settlement-side fees (Referral, Affiliate, FBT Fulfillment, Chargebacks, Violation, Refund Admin, Campaign, Co-funded Promo, Smart Promo, all reimbursements + adjustments) The day TikTok processed the order/adjustment through settlement (Statement date column) Per Lindsay/Jack 2026-06-25 — matches when TikTok actually recognized the transaction. Under this scheme, orders placed at end of month but settled next month appear in next month.
Shipped Date Cost to Ship to Customer The day Jetpack (3PL) actually shipped the package (from Seller Shipping CSV shipped_date) Per Lindsay 2026-06-25. The 3PL invoice hits when they ship, not when the customer ordered.
Services Month (flat-spread) 12 FBT detail lines (Hub Placement, Storage, Inbound Shipping, Incidents, Booking/Routing/Outbound N/C, Delayed Response, Disposal, Return Shipping VAS, Return-to-Seller Handling, Inbound Return Operation), Cost to Ship to FBT (Jetpack invoices), all manual Monthly Input lines (Team Spend, Software, Retainers, Outsourced Agency, Off-Platform, Other G&A, TT Promo Credits fallback) The month the service was performed / period the invoice covers. Flat-spread across all days of that month. FBT detail: reverted from Statement Date to Services Month on Jack's 2026-07-01 request — matches TikTok's own billing-period grouping in the FBT portal. Manual entries: they represent recurring monthly costs so services month is the natural period.
Daily (cash) Ad Spend, TBSM Savings, TT Promo Credits (when Feed P&L is on) The date the ad was billed (Campaign Overview file) or the day the FIFO ledger drew from a discounted layer Ad transactions are billed daily by TikTok; no "statement" concept applies. TBSM discounts happen per-draw, so they align to the day of spend.
Historical note. FBT detail lines were on Statement Date methodology from 2026-06-26 to 2026-07-01 (Payment Cycle upload determined the destination month). Jack requested a revert on 2026-07-01 — the FBTBillingSchedule + Payment Cycle importer are still in the codebase (untouched) in case we need to switch back, but they no longer drive attribution.

7. COGS — SKU cost lookup

The COGS page is a table of SKU ID → cost-per-order. When Manage Orders is imported, each order's COGS is stored as quantity × COGSItem.cogs_per_order.

  • Seed: seed_cogs management command runs on every deploy — creates COGS rows for known SKUs with default costs. Never overwrites edited values.
  • Update a cost: edit inline on the COGS page. New imports use the new value. Existing Order rows keep their imported COGS — to recompute, wipe Orders and re-import.
  • New SKU: add to seed_cogs.py or insert via Django shell.

8. Ad Discounts ledger — FIFO reconciliation

/ad-discounts/ reconstructs the daily TikTok ad-account balance from Transaction and Ad Spend exports. Answers: was today's spend funded by discounted credit (and at what rate), or was it card-billed at full price?

  1. Credit layers: "Increase balance" rows in the Transaction "Others" sheet become TBSM layers (default 6% discount, overridable per date). Promotions rows are free (100% discount).
  2. FIFO drawdown: daily ad spend draws from the oldest layer first. Empty layers → next. When no layers remain → card-billed at full price.
  3. P&L integration: when Feed P&L is on, the ledger's daily savings_tbsm and savings_promo populate the corresponding P&L lines. When off, TBSM Savings falls back to $0 and TT Promo Credits comes from Monthly Input.
Known open item (2026-07): the ledger's total closing balance ($316k for Jun 30) doesn't match TikTok's Promotions tab total ($10,929). Likely causes: stale AdLedgerConfig.opening_balance or duplicate credit inflows from multiple Transaction file uploads. The per-day savings numbers are still trusted because they compute from actual drawdowns. Balance reconciliation is on the todo list.

9. P&L line items — source & formula for every row

Each line: source, formula, methodology, notes. Anything with a Statement Date, Shipped Date, Services Month, or Daily badge tells you the attribution.

REVENUE

Gross Sales Manage Orders Statement Date
From: SKU Subtotal Before Discount from Manage Orders, joined to Settlement by order_id.
Formula: sum, grouped by MIN(statement_date) across the order's Settlement rows.
Why MIN over all history: guarantees the order attributes to ONE canonical statement date regardless of which date range the query asks for. Prevents dashboard/export drift where the same order appeared in different months depending on view.
Watch: orders placed but not yet settled don't appear until a fresh Settlement file is uploaded. Includes canceled orders — the Settlement-side refund line nets them out to $0.
Less: Promos & Discounts Manage Orders Statement Date
From: SKU Seller Discount from Manage Orders, joined to Settlement.
Formula: -1 × sum, grouped by statement date.
Only seller discount is captured: Platform discounts are TikTok-funded — they don't reduce your revenue. Your Gross Sales already includes the pre-Platform-Discount amount (TikTok reimburses you separately), so booking Platform discounts would double-count.
GMV CALC
Formula: Gross Sales + Less: Promos & Discounts.
Method A — computed from Manage Orders (accrual-consistent with the rest of the P&L). TikTok's Shop Analytics GMV shows separately as a reference for comparison.
GMV (TikTok Analytics — reference) Shop Analytics
Formula: as-reported by TikTok, no math applied.
REFERENCE ONLY. Doesn't drive Net Revenue. Small differences vs computed GMV are expected (TikTok filters some order types internally).
Less: Refunds Settlement Statement Date
From: Settlement, Gross sales refund + Seller discount refund columns.
Formula: sum by statement date. Stored negative.
Captures both: pre-ship cancellation refunds and post-delivery return refunds. Since Gross Sales includes canceled orders, this line nets them out.
NET REVENUE CALC
Formula: GMV + Less: Refunds.
Every % column in the P&L is computed against this denominator.

COGS

COGS Manage Orders × COGS Statement Date
From: per-order COGS (qty × per-SKU cost) summed by statement date.
Formula: sum for orders that are (a) non-canceled OR (b) canceled-but-shipped (identified via FBT Fulfillment Fee row in Settlement).
Canceled-but-shipped: if the order shipped before it was canceled, the fulfillment cost was already incurred — COGS stays. If canceled pre-ship, COGS is $0.

10. How the numbers roll up

  1. Daily P&L (compute_daily_pnl in aggregator.py) for a date range:
    1. SQL CTE joins Order to SettlementRow: computes MIN(statement_date) over ALL history per order, filters at outer query. Sums Gross / Promos / COGS per statement_date.
    2. Aggregates SettlementRow by statement_date: refunds, all fees, adjustments.
    3. Overrides GMV (Analytics reference) from AnalyticsDay.
    4. Sets raw Ad Spend from AdSpendDay.
    5. Flat-spreads all Monthly Input fields across their services month (FBT detail + Cost to Ship to FBT + all manual entries).
    6. Overrides Cost to Ship to Customer with per-day sums from SellerShipmentCost grouped by shipped_date.
    7. If AdLedgerConfig.feed_pnl is on, overlays TBSM Savings + TT Promo Credits from AdLedgerDay.
    8. Computes derived rows: GMV, Net Revenue, Gross Profit, Total Ad Spend, Total Marketing, Total G&A, Net Profit.
  2. Monthly P&L (compute_monthly_pnl) runs ONE year-wide compute_daily_pnl, buckets results by month.
  3. Cache key: daily_pnl:v{ImportLog.id}:{start}:{end}. Any import (including monthly_inputs_edit) bumps the version → cache invalidates → next read recomputes.
  4. Render: views.py builds rows in PNL_ROW_LAYOUT order (canonical row list with section / sub / row / total / blank types).

11. Deploying code changes

  1. Clone: git clone https://github.com/Harambe99/pnl-rosabella.git
  2. Make changes.
  3. If models changed: python manage.py makemigrations + commit the migration.
  4. git add && git commit && git push origin main
  5. Render auto-deploys on push (~3-10 min build). The Procfile's release: command runs migrations + seed_cogs + collectstatic.
  6. Verify: curl https://pnl-rosabella.onrender.com/health/ → OK
  7. If deploy seems stuck: check Render dashboard → Deploys tab for build logs. Build errors show there.

Local dev: pip install -r requirements.txt, set APP_PASSWORD and DATABASE_URL, python manage.py migrate, python manage.py seed_cogs, python manage.py runserver.

13. For the next maintainer / AI agent

  1. Read this README top-to-bottom, especially section 4 (methodology).
  2. Skim core/models.py, core/importers.py, core/aggregator.py — those three files contain the vast majority of logic.
  3. Before adding any new P&L line from a Settlement column: grep for prior handling / removal. TikTok has many overlapping columns (parents + sub-components), and adding one that's already inside another is the most common footgun. Confirmed examples:
    • Platform discounts — TikTok-funded, already inside Gross Sales via SKU Subtotal Before Discount. Do NOT add.
    • Sales tax payment/refund — marketplace facilitator pass-through. Do NOT add.
    • Seller Shipping Fee Discount — removed 2026-06-30, was already inside NET Customer-Paid Shipping Fee.
  4. Cache invalidation contract: any DB mutation that affects P&L must write an ImportLog row so _data_version() bumps. See the Monthly Inputs edit view for the pattern.

14. Known gotchas & design decisions

  • Settlement files MUST be parsed with pandas, not openpyxl read-only. openpyxl's read-only mode has a bug that returns only ~5 rows from the iterator for these files.
  • MIN(statement_date) is computed over ALL history, not the query range. Without this, the same order attributes to different months depending on which range you query — dashboard and export drift by a few $k per month. Fixed on 2026-07-01.
  • Sales tax is a pass-through. TikTok is a marketplace facilitator in all 45 US sales-tax states. Sales tax columns in Settlement are informational, not P&L-relevant.
  • Platform discounts are TikTok-funded, not seller-funded. Our Gross Sales includes the full pre-Platform-Discount amount (SKU Subtotal Before Discount). TikTok reimburses the seller via a separate Settlement mechanism. Booking Platform discounts as revenue would double-count ~$477k across 8 months. Documented in Manage Orders "columns we deliberately ignore".
  • Seller Shipping Fee Discount was removed. It's already inside NET Customer-Paid Shipping Fee (the column we book). Booking it separately double-counted the seller's shipping absorption. Verified on May 2026 data: our current sum of 8 sub-components matches TikTok's Shipping parent to the penny.
  • Ad Transactions has TWO supported formats. Legacy: 3-sheet workbook (Payments / Promotions / Others). New: single sheet with a "Transaction type" column that we auto-route. Both accepted.
  • Seller Shipping CSV has TWO supported formats. Legacy: only reference_number. New: distinct shipment_number and reference_number columns. When a new-format file is uploaded, the importer auto-deletes any old-format duplicates for the overlapping period.
  • Intermittent 502s under heavy load. Gunicorn is 2 workers × 4 threads. A heavy Settlement import (7+ min) + concurrent dashboard requests can exhaust worker slots for a few seconds. Requests fail 502 during the window but the app recovers quickly. Mitigation: avoid loading pages during long imports.
  • LocMemCache is per-worker. Each Gunicorn worker builds its own cache. With 2 workers, cache is fragmented but each is version-keyed by ImportLog.id, so stale reads are impossible after any import.
  • Ad Ledger balance ≠ TikTok UI balance (as of 2026-07). Ledger shows $316k, TikTok's Promotions tab shows $10,929. Likely stale AdLedgerConfig.opening_balance or duplicate credit inflows. The daily savings numbers are still trusted since they compute from actual drawdowns. Balance reconciliation is on the todo list.
  • Nov + Dec 2025 All Order CSVs cannot be downloaded from Seller Center (redirect to error page). Nov/Dec income files ARE uploaded and provide Settlement fees. But Nov/Dec P&L Gross/Promos/COGS are incomplete since no Order rows exist for those months.
  • FBT Payment Cycle importer is deprecated for P&L. The FBTBillingSchedule model + Payment Cycle import path still exist. Uploading Payment Cycle data won't change any P&L line — kept purely as a "just in case" for a future methodology switch.
  • Section / sub / blank rows render empty cells, NOT dashes. Dashes only appear on data rows with value 0 or null.
  • The /wipe/ endpoint is CSRF-exempt so it can be hit via curl. Still gated by the shared-password middleware.

If you find something in the code or docs that contradicts the live behavior, the code is the source of truth — please update this README so the next person isn't misled. Last major revision: 2026-07-02.
